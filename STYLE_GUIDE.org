#+TITLE: nexenne C++ style guide
#+SUBTITLE: the style every file in this repo follows
#+OPTIONS: toc:2 num:nil ^:nil
#+STARTUP: showall

The goal: open any file and know what the next line looks like before reading
it. =.clang-format= handles layout, =.clang-tidy= enforces the rules below.
When this guide and the tooling disagree, the tooling wins; fix both together.

*Indent with two spaces*, never tabs; wrap at 100 columns. Always braced
initialization, never === init. Name stored types with =using= aliases and use
the alias, not the raw type.

* Language

C++23, =CXX_EXTENSIONS OFF=. Reach for the modern facilities: concepts,
ranges and =std::views=, =std::string_view= / =std::span=, =std::expected=,
=if constexpr=, =[[nodiscard]]=.

* Naming

snake_case for everything. The only suffix/prefix rules:

| construct                  | rule        | example                    |
|----------------------------+-------------+----------------------------|
| types, functions, members  | snake_case  | =vector_view=, =to_string= |
| private / protected member | =m_= prefix | =m_count=                  |
| type template parameter    | PascalCase  | =Real=, =Value=, =T=       |
| value template parameter   | PascalCase  | =N=, =Width=, =Capacity=   |
| metafunction alias result  | =_t= suffix | =tl_concat_t=, =value_t=   |
| member type alias          | =_type=     | =value_type=, =storage_type= |
| variable template          | =_v= suffix | =pi_v=, =is_unit_v=        |
| macro                      | UPPER_CASE  | =NEXENNE_MATH_VERSION=     |

Template parameters are the one exception to snake_case: they are PascalCase,
matching standard-library style (=std::vector<T, Allocator>=). Everything else
is snake_case, never camelCase, never a trailing underscore (=count_= is
wrong). Public members carry no prefix.

The =_t= suffix follows the standard library and is *reserved for the result of
a metafunction alias template*, =using foo_t = typename foo<...>::type;= (the
analogue of =std::remove_cv_t= or =std::tuple_element_t=). A *concrete class,
struct, or enum takes no suffix* (=sin_cos=, =vector=, =matrix=, never
=sin_cos_t=), exactly as =std::vector= and =std::optional= carry none. A type
alias that names a member uses the full =_type= (=value_type=, =storage_type=),
matching =std::value_type=; a variable template uses =_v= (=pi_v=). Dimensioned
math aliases attach the dimension and keep an explicit scalar suffix:
=vector2<T>= / =vector3_f= / =matrix4_d=, never =vector_2_f= and never the
abbreviated =vec3=.

* Class layout

Order the access sections like this: a leading =public:= for type aliases and
constants that callers need (the exception), then =private:= for data members
and helpers, then =protected:=, then the =public:= interface (constructors,
methods, operators). Do not push data members to the bottom; only
externally-needed type definitions lead the class.

Name stored types with =using= aliases (for example =using function_type = Fn=)
and use the alias throughout the class, not the raw template parameter.

*Every value or wrapper type exposes a =using value_type = ...;=* as the first
entry of its leading =public:= section, naming the element or underlying type it
holds (the component type of a vector or box, the scalar of a strong type, the
stored =T= of a container). It is mandatory, not optional: generic code and the
formatters rely on it, and a type that prints its state should never force a
caller to rediscover the element type. Use the alias internally too rather than
respelling the template parameter. Other member aliases take the =_type= suffix
(=size_type=, =const_reference=); only the metafunction-result conventions in the
naming section differ.

Alias *any* long, complex, or repeated type expression and use the alias rather
than spelling it inline: function-pointer types as =using fn = R (*)(Args...)=
(never a name buried mid-declarator like =R (*p)(Args...)=), and nested
template-ids like =std::common_type_t<value_t<A>, value_t<B>>= as a named
=detail= alias. Never write the same complex type inline more than once.

* Special members and operators

- Prefer the *rule of zero*: declare no special members when the compiler's
  defaults are correct. Reach for them only when ownership demands it.
- When a type needs assignment, write one unifying =operator=(T value)= by
  value (copy-and-swap); the compiler routes both copy and move through it. Do
  not hand-write separate copy and move assignment.
- For ordering, define =operator<=>= (and =operator=== when needed) and let the
  compiler generate the other comparisons. Never hand-write =<=, =>=, =<==.
- To make a type non-copyable and non-movable, delete the copy constructor and
  the single copy =operator==; the move operations are then suppressed.
- A scope-bound RAII guard (=defer=, =scope_guard=) is =[[nodiscard]]= (a
  discarded temporary, which would fire immediately, warns) and =final=, and is
  non-copyable and non-movable by the rule above.
- Where a move-assignment must re-establish a member whose type is not
  move-assignable (a captured lambda has no =operator==), reconstruct it with
  =std::destroy_at= + =std::construct_at= rather than assigning.

* Functions

Trailing return type, always:

#+BEGIN_SRC cpp
[[nodiscard]] constexpr auto square(real_t const value) noexcept -> real_t;
#+END_SRC

- =[[nodiscard]]= on pure functions that return a value.
- =noexcept= when it never throws; =constexpr= / =consteval= when usable at
  compile time.

Accessors and returns:

- No =get_= / =set_= prefixes: an accessor is =position()=, a mutator is
  =position(p)=. Keep verb prefixes for actions.
- Return =T const&= to read a non-trivial member (or =T= by value when cheap);
  do not hand out a non-const reference to internal state unless mutation
  through it is intended.
- Add ref-qualifiers where it matters: =value() const&= returns =T const&=,
  =value() &&= returns =T&&= so callers can move out of an rvalue.

Parameter passing:

| kind                           | pass as                          |
|--------------------------------+----------------------------------|
| trivially copyable, =<=2 words | by value, =const=                |
| string / range view            | =std::string_view= / =std::span= |
| ownership transfer             | by value or =T&&=                |
| large, read-only               | =T const&=                       |
| out / in-out scalar            | =T*= (never =T&=)                |
| caller-owned buffer to fill    | =T&= or =std::span=              |

* Initialization and const

Braced init only. Every value not reassigned is =const=. East const.

#+BEGIN_SRC cpp
auto const g{real_t{-9.81}};
auto const hit{std::ranges::find_if(items, pred)};
auto find(grid_t const& g, point_t const& p) -> cell_t const*;
#+END_SRC

Forbidden: =T x(...)=, =auto x = expr=, =int x = 0=. Members default with
=T m_x{};=. Loops too: =for (auto i{0}; ...)=. Non-mutating methods are =const=.

* Idioms and performance

- *Reuse the standard library; do not reinvent it.* Use what is already in
  =std=, the numeric constants in [[https://en.cppreference.com/w/cpp/header/numbers][=<numbers>=]] (=std::numbers::pi=, =e=,
  =sqrt2=, =ln2=, ...), the algorithms in =<algorithm>= / =<numeric>= /
  =<ranges>=, the type traits in =<type_traits>=, and the utilities in
  =<bit>= / =<cmath>= / =<concepts>=, instead of hand-rolling an equivalent.
  Re-implement a standard facility *only* for a strict, stated reason, and say
  which in a comment: a measured optimization (a fast approximation, a SIMD
  layout), a guarantee the standard one lacks (=constexpr=, =noexcept=,
  deterministic rounding, freestanding/no-allocation), or a genuine gap in the
  standard. A module's own constants header (for example =math::constants=)
  *forwards* the =std= values and only *adds* what is missing; callers reuse it
  rather than typing a literal.
- Prefer =std::ranges= algorithms and =std::views= over raw loops.
- =std::array= over C arrays; smart pointers over =new= / =delete=.
- Never a C-style =(T)x= or functional =T(x)= cast: use =static_cast= (and
  =reinterpret_cast= / =const_cast= when unavoidable), or =static_cast<void>(x)=
  to discard a value.
- Mark =constexpr= / =noexcept=; avoid needless allocations; =std::move= only
  on a genuine ownership transfer (never a =const= or a return value).
- Mark a potentially-empty stored member (a stateless deleter, comparator, or
  allocator) =[[no_unique_address]]= so it costs no space.
- Debug-only contract checks: =assert(cond && "msg")= on a documented =@pre=;
  the precondition is the caller's to uphold and is undefined behaviour (not an
  exception) in release.

* File header

Headers open with =#pragma once=, then a =/** @file @brief */= block; source
files open with the same block. Licence and authorship live in the LICENSE file
and git, never in a per-file banner.

#+BEGIN_SRC cpp
#pragma once

/**
 * @file
 * @brief Strong typedef wrapper for distinct integer-like types.
 */

#include <concepts>
#+END_SRC

* Headers and namespaces

- Include order (clang-format regroups): doctest, std, third-party, =<nexenne/...>=,
  local.
- Every public symbol is in =nexenne::<module>=. Implementation details go in a
  nested =detail= namespace, or an unnamed namespace when the helper must not
  leak out of the translation unit.
- Constrain every template with a concept, and prefer a concept or =requires=
  over a =static_assert=. Template parameters are PascalCase (=Real=, =Value=,
  =N=). Include rather than forward-declare unless it breaks a cycle.
- *Name concepts the way the standard library does* (=std::integral=,
  =std::movable=, =std::same_as=): no =Is=/=Has= prefix, no =_concept= suffix,
  and the name reads as a predicate on the type. Pick the form by what the
  constraint asserts:
  - *category membership* : a plain noun or adjective (=arithmetic=,
    =scoped_enum=, =rng_engine=), mirroring =std::integral= / =std::range=.
  - *a capability* : an =-able= adjective (=hashable=, =blockable=), mirroring
    =std::swappable= / =std::sortable=.
  - *structural conformance to an archetype* : the archetype name plus a =_like=
    suffix (=clock_like=, =pointer_like=), mirroring C++23 =tuple-like= /
    =pair-like=.
  - *a relation between two or more types* : a relational suffix =_as= / =_to= /
    =_from= / =_with= / =_for= (=same_tag_as=), mirroring =std::same_as= /
    =std::convertible_to= / =std::assignable_from=.
  A role-noun concept may carry its role (=checkpointable_arena=,
  =transparent_comparator=) the way =std::input_iterator= does. Test-local
  =requires= probes follow the same rule.

* Error handling

- *=std::expected<T, error_t>= for every recoverable error.* Never throw to
  report one. Throwing is allowed only where it is the single correct option by
  design: a constructor that cannot otherwise signal failure, or allocation
  (=new=). Everything else returns =std::expected=.
- =assert= for internal preconditions in Debug, never on release input.
- Document =@pre= / =@post= on public functions.

* Formatting and diagnostics

*Every public type ships a formatter so its state prints with no caller-written
helper.* If a user would otherwise write a =for= loop or a hand-rolled string to
inspect a value, the type is missing its formatter. For each public value type
(and each public enum, errors included) provide the three layers, in order:

- =to_string(x)= returning a readable =std::string=,
- =operator<<(std::ostream&, x)= that streams =to_string=,
- a =std::formatter<X>= specialization so =std::format("{}", x)= works.

Place these in the module's =format.hpp=, never in the type's own header and
never in the light =error.hpp=: =<format>= is heavy and must stay out of hot and
widely-included headers, so formatting is opt-in by including =format.hpp=.

- *Enums* (an =error= enum returned from =std::expected=, say) inherit the string
  formatter and forward to =to_string=, so a spec applies and no manual
  =to_string= call is needed at the call site:
  #+BEGIN_SRC cpp
  template <>
  struct std::formatter<module::some_error> : std::formatter<std::string_view> {
    template <typename Ctx>
    auto format(module::some_error const e, Ctx& ctx) const {
      return std::formatter<std::string_view>::format(module::to_string(e), ctx);
    }
  };
  #+END_SRC
- *Match the house output style* so the module reads uniformly: sequences as
  =[a, b, c]=, set-like as ={a, b, c}=, map-like as ={k: v, ...}=. A composite
  type forwards a parsed spec to each element's formatter (see the math vector
  and matrix formatters).
- *Do not wrap standard types.* =std::vector=, =std::array=, =std::map=,
  =std::pair=, =std::tuple=, =std::span= and friends are already formattable in
  C++23; rely on the standard rather than adding a formatter for them.
  (=std::optional= / =std::expected= formatting is C++26, not yet available.)
- The =format= (and =parse=) method is a public symbol: give it the full Doxygen
  block, since the docs build fails on warnings.

* Comments

- *No decorative banner comments* (=// -----=, =# =====). Use a plain short
  comment or a Doxygen block, nothing else.
- *No em-dash, no =--= em-dash substitute, and no backticks*, in comments,
  strings, and prose docs alike. Use a comma, a colon, or parentheses for an
  aside; use Doxygen =\c= for inline code in a comment.
- *No HTML* in comments.
- In function bodies, comment the /why/, never the /what/. Default to none.

** Doxygen on every public symbol

Javadoc-flavored =/** ... */=, no HTML, no backticks (use =\c= for an
identifier, =\p= for a parameter). The block is always built from the same
ordered groups, one blank comment line between groups:

1. =@brief=, one line.
2. Detail paragraph.
3. =@tparam= then =@param=.
4. =@return=, alone (a blank line both before and after it).
5. =@pre=, =@post=, =@throws=.
6. =@note=, =@warning=, =@see=, =@complexity=, =@par Example=.

Always spell =@brief= out. =@pre= and =@post= are mandatory, even when the
answer is "None". A =void= function has no =@return= (Doxygen rejects it).
Drop any /optional/ tag that only restates the signature. Put exactly one space
after every tag (and after a =@param= / =@tparam= name): no column alignment.
The order never varies between files; the example below is exact.

The docs build runs under =FAIL_ON_WARNINGS= with =WARN_NO_PARAMDOC=, which adds
three hard rules:

- *Every documented entity with parameters or a return needs =@param= /
  =@return=, operators included.* A one-line =/// @brief= on an operator that
  takes parameters fails the docs build; give every operator the full block.
- Hide implementation that must not appear in the API docs (a =detail= helper,
  or a self-referential primary template Doxygen misreads as a "recursive class
  relation") between =/// @cond INTERNAL= and =/// @endcond=.
- Never leave a bare =<word>= token in Doxygen text: Doxygen reads =<bit>=,
  =<I>=, =<utility>= as HTML tags and the build rejects them. Drop the angle
  brackets or use =\c=.
- *One =/** */= block per symbol; never stack =///= lines.* A function, method,
  accessor, constructor, class, or alias is documented with a single multi-line
  =/** ... */= block, even a one-line accessor (its =@brief= and =@return= go on
  separate lines inside the block). The only short form is a trailing
  =///< one line.= on an enumerator or data member: a single fact, no tags. Two
  or more =///= lines, or any =///= carrying =@brief= / =@param= / =@return=, is
  wrong, rewrite it as a =/** */= block. So this is forbidden:
  #+BEGIN_SRC cpp
  /// @brief Signed plane offset.
  /// @return Const reference to the stored offset.
  [[nodiscard]] constexpr auto offset() const noexcept -> Real const&;
  #+END_SRC
  and this is the rule:
  #+BEGIN_SRC cpp
  /**
   * @brief Signed plane offset.
   *
   * @return Const reference to the stored offset.
   */
  [[nodiscard]] constexpr auto offset() const noexcept -> Real const&;
  #+END_SRC

#+BEGIN_SRC cpp
/**
 * @brief Kinetic energy of a body of mass \p mass at velocity \p velocity.
 *
 * Implements \c E_k = 1/2 * m * v^2; the result is non-negative.
 *
 * @tparam Real Floating-point type satisfying \c std::floating_point.
 * @param mass Mass in kilograms.
 * @param velocity Velocity in metres per second.
 *
 * @return Kinetic energy in joules.
 *
 * @pre \p mass is finite and non-negative.
 * @post Returned value is finite and non-negative.
 *
 * @complexity \c O(1).
 */
template <std::floating_point Real>
[[nodiscard]] constexpr auto kinetic_energy(Real const mass, Real const velocity)
  noexcept -> Real;
#+END_SRC

=@complexity= and =@thread_safety= are project aliases (defined in the
Doxyfile); use =@complexity= on anything non-trivial.

* A complete example

Every rule above in one small type: trailing returns, =const= by default, braced
init, the rule of zero, a defaulted =operator<=>= plus =operator===, =constexpr=
throughout, and =std::expected= instead of throwing.

#+BEGIN_SRC cpp
enum class counter_error { overflow };

/**
 * @brief A counter that reports overflow instead of wrapping.
 *
 * @tparam T Unsigned integral counter type.
 *
 * @pre None.
 * @post A default-constructed counter is zero.
 */
template <std::unsigned_integral T>
class saturating_counter {
public:
  using value_type = T;

private:
  value_type m_value{};  // braced default member init

public:
  // Rule of zero: the compiler's special members are correct, so none are declared.

  /**
   * @brief The current count.
   *
   * @return The stored value.
   *
   * @pre None.
   * @post None.
   */
  [[nodiscard]] constexpr auto value() const noexcept -> value_type {
    return m_value;
  }

  /**
   * @brief Adds \p amount, reporting overflow instead of wrapping.
   *
   * @param amount Amount to add.
   *
   * @return The new count, or \c counter_error::overflow when the addition
   *         would exceed the maximum of \p T.
   *
   * @pre None.
   * @post On success the count has increased by \p amount.
   */
  [[nodiscard]] constexpr auto add(value_type const amount) noexcept
    -> std::expected<value_type, counter_error> {
    if (amount > std::numeric_limits<value_type>::max() - m_value) {
      return std::unexpected{counter_error::overflow};  // no throw
    }
    m_value = static_cast<value_type>(m_value + amount);
    return m_value;
  }

  /// @brief Equality of two counters; the compiler derives \c != from it.
  [[nodiscard]] friend constexpr auto
  operator==(saturating_counter const&, saturating_counter const&) noexcept -> bool = default;

  /// @brief Ordering of two counters; the compiler derives \c < \c > \c <= \c >=.
  [[nodiscard]] friend constexpr auto
  operator<=>(saturating_counter const&, saturating_counter const&) noexcept = default;
};
#+END_SRC

* Naming files

- Directories and files: =lower_snake_case=. Headers =.hpp=, sources =.cpp=.
- Tests: =test_<topic>.cpp= under =tests/=.
- Examples: =examples/<module>/<topic>.cpp=, one runnable =main= per class.
- Module =foo= umbrella header is =include/nexenne/foo/foo.hpp= and includes
  every leaf header.
- All repo documentation (guides, READMEs, this file) is org-mode (=.org=),
  never markdown. Per-module usage guides live in =doc/module/<name>/=.
- Never put a =[[...]]= link inside an org table cell: the markup width is not
  the rendered width, so any realign corrupts the columns. List link-bearing
  entries as =- [[link]] : description= instead; keep tables for plain text.
- Documentation is a hierarchy of indexes: the top index links to each
  subfolder index, which links to its articles. Each level is shaped for its
  own content.

* Tooling

=clang-format= and =clang-tidy= (both version 18+) read the configs at repo
root. A change is not done until format, tidy, the warnings-as-errors build,
the sanitizers, and the docs build are all clean:

#+BEGIN_SRC sh
clang-format -i $(git ls-files '*.hpp' '*.cpp')
run-clang-tidy -p build/dev -quiet
cmake --preset ci   && cmake --build --preset ci   --target nexenne_tests  # warnings as errors
cmake --preset asan && cmake --build --preset asan --target nexenne_tests  # ASan + UBSan
cmake -S . -B build/docs -DNEXENNE_BUILD_DOCS=ON \
  && cmake --build build/docs --target nexenne_<mod>_docs                  # docs: FAIL_ON_WARNINGS
#+END_SRC
