#+STARTUP: showall

The full, enforced rules live in [[file:../../STYLE_GUIDE.org][STYLE_GUIDE.org]]
(mechanically enforced by [[file:../../.clang-format][.clang-format]] and
[[file:../../.clang-tidy][.clang-tidy]]) and the workflow rules in
[[file:../../CONTRIBUTING.org][CONTRIBUTING.org]]. This page is an orienting
summary, not a substitute; read those for the detail. The hooks
([[file:hooks.org][Git hooks]]) and [[file:ci.org][CI]] enforce the formatting
and lint rules on every change.

* Code

- C++23, =snake_case= for every identifier; template parameters are =CamelCase=
  (for example =Real=, =N=). Private and protected members take the =m_= prefix.
  These names are checked by =readability-identifier-naming= in =.clang-tidy=.
- The =_t= suffix follows the standard library: it marks the *result of a
  metafunction alias template* (=using foo_t = typename foo<...>::type=, like
  =std::remove_cv_t=). A concrete class, struct, or enum carries *no suffix*
  (=sin_cos=, not =sin_cos_t=, just as =std::vector= has none); a member type
  alias uses =_type= (=value_type=); a variable template uses =_v= (=pi_v=).
- Trailing return types everywhere (=modernize-use-trailing-return-type=), braced
  init, =const= by default (=misc-const-correctness= transforms non-reassigned
  locals), ranges over raw loops.
- =std::string_view= / =std::span= for non-owning arguments.
- East const, pointer and reference bind to the type (=int const* p=), two-space
  indent, 100-column lines, attached braces, LF line endings. These come from
  =.clang-format=.
- Comments: plain or Doxygen only. No banner comments, no em-dash, no backticks,
  no HTML.

* File header

#+BEGIN_SRC cpp
#pragma once

/**
 * @file
 * @brief One-line description.
 */
#+END_SRC

No SPDX line and no per-file author list: the licence is in
[[file:../../LICENSE][LICENSE]], authorship is in git.

* Doxygen blocks

Fixed tag order, every block: =@brief=, detail, =@tparam= / =@param=, =@return=,
=@pre= / =@post= / =@throws=, then =@note= / =@warning= / =@see= /
=@complexity= / =@par Example=. =@pre= / =@post= are mandatory. The doc build
enforces correctness of these tags, not their presence (see
[[file:api-docs.org][API docs]]); the canonical block is in
[[file:../../STYLE_GUIDE.org][STYLE_GUIDE.org]].

* Commits

Subject line is =type(scope): description=, for example
=feat(utility): add strong_typedef=. The scope is required (a module name or an
area such as =cmake=, =docs=, =ci=, =template=, =hooks=, or =repo=). One logical
change per commit, no body, no =Co-Authored-By= or AI trailer. The type table and
full rules are in [[file:../../CONTRIBUTING.org][CONTRIBUTING.org]].

* Docs format

All prose docs are Org mode (=.org=), never Markdown; do not add =.md= files.

* Documenting the math-heavy modules

The numerically dense modules, =math= and =geometry= (and the tower above them,
=physics= and =simulation=), carry a heavier documentation burden than the rest
of the library: the techniques are hard to follow from the code alone and hard to
search for online. They are documented to a stricter standard, and any new code
in them must meet it.

- *Explain every non-obvious technique in the code itself*, not only in the
  =.org= article. A reader of the header should understand the /why/ and /how/
  without leaving the file. This covers bit-tricks, range reduction, polynomial
  coefficients and the interval they are fit on, Newton and Horner steps, magic
  constants, fixed-point scaling, and the quaternion, matrix, and projection
  identities.
- *Give the derivation or intuition*, not just a restatement. For example: why
  =0x5F375A86= halves the IEEE-754 exponent; why the sincos quadrant table maps
  =k mod 4= the way it does; how the Q-format =scale^2= cancels in a fixed-point
  multiply.
- *Put a searchable reference in the code comment*: an author, year, and title; a
  named identity; or a URL. References belong in the code so the technique can be
  looked up, and they are repeated in the article's =Further reading= section.
  Every math and geometry article ends with =Further reading= linking the
  authoritative sources (papers, cppreference, the named method).
- *State what a type can represent*, with the formula and concrete numbers, and
  make it queryable where it makes sense (for example =fixed='s =min()=,
  =max()=, =resolution()=, and =integer_bits=).

The aim is that both the code and the articles stand on their own and can be
picked up cold in a later session.
