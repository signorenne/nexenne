#+TITLE: ring_buffer
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::ring_buffer<T, N>= is a fixed-capacity circular FIFO queue
with inline storage. It never allocates, and it makes the overflow decision
explicit: a push into a full buffer either fails (=push=) or evicts the oldest
element (=push_overwrite=). The capacity =N= must be at least one.

* The idea

Producer/consumer queues and rolling windows want a bounded buffer that recycles
its storage instead of growing. A ring buffer lays =N= slots in a circle and
tracks a =[head, tail)= window over them: pushing advances =tail=, popping
advances =head=, and both wrap around the end. No element ever moves while it
sits in the buffer, so push and pop are =O(1)= with no allocation, and the same
=N= slots serve an unbounded stream of values.

* How it works

** Memory layout

Storage is an inline array of =N= slots (each a union of a trivial byte or a
live =T=, so building the array constructs no =T=). Three indices track the
window: =head= (the oldest element), =tail= (where the next push lands), and
=size=. Capacity is the constant =N=; there is no pointer and no heap.

#+BEGIN_EXAMPLE
ring_buffer<int, 4> after push 10, 20, 30, then one pop:

  slot:      0      1      2      3
          +------+------+------+------+
          | ~~~~ |  20  |  30  | ~~~~ |   ~~~~ = empty slot
          +------+------+------+------+
                    ^             ^
                  head=1        tail=3        size = 2

  logical FIFO order, read from head: 20, 30
#+END_EXAMPLE

** Wrap-around

=tail= and =head= step forward by one and wrap back to slot 0 at the end. The
wrap never uses a modulo: when =N= is a power of two it is a single mask
(=i & (N - 1)=), and otherwise a compare (=i >= N ? i - N : i=, valid because
every index is in =[0, 2N)=). Either way the hot push and pop paths stay off the
division unit. That is how the buffer reuses freed slots without shifting
anything:

#+BEGIN_EXAMPLE
push 40, then push 50 into the buffer above:

  +------+------+------+------+
  |  50  |  20  |  30  |  40  |     50 went in slot 0; tail then advanced to 1
  +------+------+------+------+
            ^
        head, tail                   now full: size = 4, head == tail == 1
#+END_EXAMPLE

When the buffer is full =head == tail=: the next =push= reports
=container_error::full=, while =push_overwrite= destroys the element at that slot
(the oldest), constructs the new one in its place, and advances both indices, so
the window slides forward and the size stays =N=.

** Access and iteration

Logical indices count from the front, so =operator[](0)= and =front()= are the
oldest element and =back()= the newest, each mapping through the same =wrap=.
=operator[]= is unchecked; =at=, =front=, and =back= are checked and return a
=nullptr= pointer on a miss. The forward iterator walks =size= elements from
=head= in FIFO order, wrapping as it goes, so a range-based =for= visits oldest
to newest. Construction, the pushes, pops, and access are all =constexpr=.

=pop= returns the value by =result<T>= (it moves the front out, then destroys the
slot), so the oldest element leaves in one call; there is no separate
front-then-discard dance.

** Moves and the canonical head

Copy and =operator==-style construction rebuild the FIFO by pushing each element
in order. A move walks the source from its =head=, reconstructing the elements
into the destination starting at slot 0, so the moved-into buffer always has
=head == 0= (a canonical layout) while the source is left empty. =swap= is built
from those moves, so it is =O(N)=, not a pointer exchange.

* Complexity

| operation                       | cost     |
|---------------------------------+----------|
| push / emplace                  | O(1)     |
| push_overwrite                  | O(1)     |
| pop                             | O(1)     |
| operator[], at, front, back     | O(1)     |
| copy / move construct or assign | O(size)  |
| swap                            | O(N)     |
| clear                           | O(size)  |
| iterate begin .. end            | O(size)  |

The =wrap= behind every index is branch-light: a single mask when =N= is a power
of two, otherwise a compare-subtract, never a modulo. Push, pop, and index access
are exact =O(1)= (no growth path, so not amortised).

Memory overhead is the unused slots: the object always occupies =N * sizeof(T)=
plus three =size_t= indices (=head=, =tail=, =size=), whatever the live count.
No heap block, no per-element bookkeeping.

* When to use it

Reach for =ring_buffer= when you want a bounded queue that recycles its storage
instead of growing, and you want the overflow policy to be explicit:

- =push= reports =container_error::full= so a producer can block, sleep, or drop:
  bounded event and work queues.
- =push_overwrite= evicts the oldest element to make room, so the buffer always
  holds the most recent =N=: rolling windows, sample histories, rolling
  statistics, last-=N= command history.

It is the wrong choice when the capacity bound is unknown or must grow, when you
need random insertion or erase in the middle, or when both ends and the middle
see edits. Prefer a sibling when:

- you need an unbounded or growable sequence: [[file:small_vector.org][small_vector]] (or =std::vector=).
- you can prove a hard bound but want LIFO/random push_back rather than FIFO:
  [[file:static_vector.org][static_vector]].
- edits cluster at a moving cursor (text editing): [[file:gap_buffer.org][gap_buffer]].

Reference, pointer, and iterator rules:

- a slot never moves while its element sits in the buffer, so a reference or
  pointer from =operator[]=, =at=, =front=, or =back= stays valid until that
  element is popped or evicted (=push_overwrite= when full destroys the oldest
  slot and reuses it).
- =pop= and =push_overwrite=-on-full destroy an element, dangling any reference
  to it; the other live elements keep their addresses.
- iterators capture =head= and an offset, so a =push= or =pop= changes the
  logical window and any outstanding iterator should be treated as invalidated
  for traversal even though the underlying slots have not moved.
- a move or =swap= reconstructs elements in new slots (and canonicalises =head=
  to 0), so it invalidates all references, pointers, and iterators into both
  buffers.

* Usage

#+BEGIN_SRC cpp
nexenne::container::ring_buffer<int, 64> events;

if (auto const r{events.push(code)}; !r.has_value()) {
  drop_or_block();           // container_error::full: backlog is full
}

nexenne::container::ring_buffer<int, 3> recent;
recent.push_overwrite(sample);  // always keeps only the most recent 3

while (auto const value{events.pop()}) {
  handle(*value);            // FIFO: oldest first
}
#+END_SRC
