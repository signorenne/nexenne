#+TITLE: intrusive_list
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::intrusive_list<T>= is a doubly-linked list whose links live
inside the elements themselves: =T= derives from =intrusive_list_hook<T>= and
carries its own =prev= / =next=. The list owns nothing, so insert, erase, and
splice allocate nothing, and erasing a known element is =O(1)= with no search.

* The idea

A =std::list= allocates a node per element and, to erase by value, has to find
the node first. When the element is something you already hold a pointer to, both
costs are avoidable: store the links inside the element and the list is just
pointer surgery, with no allocator and no lookup. The trade is ownership and
exclusivity: the caller owns each element's lifetime, and since an element
carries exactly one link pair it can be in at most one list at a time.

* How it works

** Hook, sentinel, and the CRTP downcast

Each element embeds a hook, one =prev= pointer and one =next= pointer, by
deriving =T= publicly from =intrusive_list_hook<T>=. The CRTP (the hook is
templated on the deriving type) lets the iterator turn a hook pointer back into a
=T*= with a plain =static_cast=: because the hook is a base subobject of =T=, the
downcast is a compile-time address adjustment, no lookup and no stored type tag.
A fresh hook is unlinked (both links null), and =is_linked()= just reports
whether either link is non-null. The hook befriends =intrusive_list= so only the
list may touch the raw links.

The list object itself holds a single =sentinel= hook and keeps the ring
*circular* through it: an empty list points the sentinel's =next= and =prev= at
itself. =begin()= is =sentinel.next=, =end()= is the sentinel. Because the ring
closes through the sentinel, no link is ever null while linked, so iteration and
insertion never branch on null, and a boundary insert (front or back) runs the
exact same four-pointer code as a middle insert.

#+BEGIN_EXAMPLE
intrusive_list<body>: the links live inside each body, around a sentinel

  +----------+    +--------+    +--------+    +--------+
  | sentinel |<-->| body A |<-->| body B |<-->| body C |
  +----------+    +--------+    +--------+    +--------+
       ^                                          |
       +------------------------------------------+
       circular: sentinel.next = A, sentinel.prev = C
#+END_EXAMPLE

** Walking a push and an erase

=insert(it, value)= (which =push_back= and =push_front= call with =end()= and
=begin()=) splices =value= in just before =it= by rewriting *four* pointers. Let
=next= be =it='s node and =prev= be =next->prev=; the list links =prev <-> value=
and =value <-> next=, then bumps =size=. No node is allocated; the element's own
embedded hook is the node.

#+BEGIN_EXAMPLE
insert value V before next N (prev P was N's old predecessor)

  before:   ... P <----------> N ...
  after:    ... P <--> V <--> N ...
            link(P, V); link(V, N)   four pointer writes, then ++size
#+END_EXAMPLE

=erase(value)= is the payoff: given just a reference to an element, it unlinks it
in =O(1)= with *no search*, because the element already knows its own neighbours.
It links =value->prev= directly to =value->next= (closing the gap), nulls the
element's two links so it reads as unlinked, and decrements =size=. A debug
=assert= catches erasing an element that is not linked. =erase(iterator)= does
the same and returns an iterator to the following element; =pop_front= /
=pop_back= unlink the sentinel's neighbour and hand back a pointer (=nullptr=
when empty). =front= / =back= return the same nullable pointers. None of these
destroy anything: the element survives, now detached.

** Ownership, lifetime, and move

The list never allocates, copies, or destroys an element. =clear= and the
destructor walk the ring detaching every element (each becomes
=is_linked() == false=) and reset the sentinel to empty; the caller still owns
the elements. An element must therefore outlive its membership, and you must
unlink it before its storage goes away.

Move *detaches*. The list does not own nodes, so it is non-copyable but movable:
a move-construct or move-assign =steal_from='s the source ring by re-pointing the
destination sentinel at the source's first and last elements, then resets the
source to empty, =O(1)=, no element touched. But the *element's* own move is
different: a moved-to hook is left detached (its move constructor links nothing,
its move assignment nulls both links), because the list still stores the
*source's* address. Move the storage of a linked element and the list dangles at
the old address: detach the element first.

* Complexity

| operation                                  | cost |
|--------------------------------------------+------|
| push_back / push_front / insert            | O(1) |
| erase by element or by iterator            | O(1) |
| pop_front / pop_back                       | O(1) |
| front / back / size / empty                | O(1) |
| move construct / move assign / swap        | O(1) |
| clear / destructor                         | O(n) |
| operator== / operator<=>                   | O(n) |

There is no per-element memory overhead beyond the two link pointers embedded in
each element and a single sentinel hook in the list object.

* When to use it

Reach for =intrusive_list= when you already hold the elements and want
allocation-free, =O(1)= membership that you mutate by element pointer: active
sets where an object cycles between lists each frame (physics islands,
animations, pending commands), LRU chains where an element moves itself to the
head, and ECS bookkeeping without a parallel map.

=intrusive_list= versus =std::list=:

- =std::list= *allocates* a node per =push= and *owns* the element (it lives in
  that node); =intrusive_list= allocates nothing and owns nothing, the element is
  the node.
- =std::list::remove(value)= is =O(n)= (it searches); =intrusive_list::erase=
  takes a reference or iterator you already hold and is =O(1)= with no search.
- =std::list= can hold the same value type in many lists freely;
  =intrusive_list= ties each element to one hook, hence one list at a time, and
  needs =T= to derive from the hook up front.

So pick =intrusive_list= when you hold pointers to the elements anyway and want
free, immediate removal; pick =std::list= when you want a container that owns and
allocates its values and a given value may sit in several lists.

Gotchas:

- *Ownership is yours.* The list destroys nothing, not even in its destructor; it
  only detaches. An element must outlive its membership, and you must unlink it
  (or destroy the list) before freeing the element's storage, or the ring points
  into freed memory.
- *One list at a time.* The hook holds a single link pair, so pushing an element
  that is already linked corrupts both lists (the precondition is "not already
  linked", and =insert= does not check it). If an element must belong to several
  lists at once, give it several hooks, one per role, by deriving through distinct
  hook bases.
- *Moving an element detaches it.* The list stores the element's *address*. A
  moved-to element is left unlinked (so it silently drops out of any list you
  expected it in), and moving a still-linked element leaves the list pointing at
  the old, now-vacated address, a dangling ring. Detach (=erase=) before moving
  the element, or simply do not move linked elements.
- *Stable otherwise.* Elements never relocate on insert or erase, so pointers,
  references, and iterators to *other* elements stay valid across any number of
  inserts and erases; only the iterator to the element you erased is invalidated.
- *No bounds or membership check on erase.* =erase(value)= trusts that =value= is
  in *this* list (the caller knows; the list does not verify beyond a debug
  =assert= that it is linked at all). Erasing an element that belongs to a
  different list corrupts that other list.

* Usage

#+BEGIN_SRC cpp
struct body : nexenne::container::intrusive_list_hook<body> { /* ... */ };

nexenne::container::intrusive_list<body> active;
active.push_back(a_body);            // O(1), no allocation
active.erase(a_body);                // O(1), no search
for (auto& b : active) { step(b); }  // bidirectional iteration
#+END_SRC
