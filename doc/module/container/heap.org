#+TITLE: heap
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::heap<T, Compare>= is a binary heap (a priority queue) built
on a contiguous =std::vector= and the standard heap algorithms. The default
=Compare= of =std::less<T>= gives a max-heap (largest on top); pass
=std::greater<T>= for a min-heap. Push and pop are =O(log n)=, and reading the
top is =O(1)=.

* The idea

A priority queue needs the most important element fast and cheap maintenance as
elements come and go. A binary heap gives exactly that with no pointers: it is
an array where each parent outranks its children, so the top is always at index
0 and a push or pop only has to sift one element along a single root-to-leaf
path. Building on =std::vector= keeps the storage contiguous and lets the
standard =push_heap= / =pop_heap= / =make_heap= do the sifting, so there is no
hand-written loop to get wrong.

* How it works

** Array layout

The heap is just a vector read as a complete binary tree: the children of index
=i= are at =2i+1= and =2i+2=. The invariant is that every parent outranks its
children under =Compare=, which puts the winner at index 0.

#+BEGIN_EXAMPLE
max-heap {5, 4, 3, 1, 1} in the vector:

  index:  0   1   2   3   4              5            <- top, index 0
        +---+---+---+---+---+           / \
        | 5 | 4 | 3 | 1 | 1 |          4   3
        +---+---+---+---+---+         / \
                                     1   1
  children of i are at 2i+1, 2i+2;  every parent >= its children
#+END_EXAMPLE

** Operations

=push= and =emplace= append the new element to the back of the vector (the next
free leaf), then =std::push_heap= sifts it *up*: it compares the element with its
parent at =(i - 1) / 2= and swaps them while the parent loses under =Compare=,
walking root-ward until the element finds a parent it does not outrank or reaches
index 0. Only that one root-to-leaf path is touched, so the cost is =O(log n)=.

#+BEGIN_EXAMPLE
push(6) into max-heap {5,4,3,1,1}: append at index 5, then sift up

  5                 5                  6          path touched: 6 -> its
 / \               / \                / \         parent 3 (3 < 6, swap),
4   3   append    4   3    swap 6,3  4   5         then parent 5 (5 < 6, swap),
/ \             / \   /            / \   /          then index 0: stop.
1  1            1 1  6             1 1  3          siblings 4,1,1 never moved.
#+END_EXAMPLE

=pop= does the mirror image: =std::pop_heap= swaps the top (index 0) with the
last element, which logically removes the winner, then sifts that last element
*down* by repeatedly swapping it with its larger-ranking child (=2i+1= or =2i+2=)
until both children lose or it becomes a leaf, restoring the invariant over the
shrunk range. =pop= then reads the now-last slot, =pop_back='s it, and returns it
as =result<T>= (or =container_error::empty= when there is nothing to pop). Both
operations are =O(log n)= because a sift visits one path of height =log n=.

Constructing from a list or range runs one =std::make_heap=, which heapifies
bottom-up in =O(n)=, strictly cheaper than =n= separate pushes (=O(n)= versus
=O(n log n)=). =rebuild= re-runs =make_heap= to re-heapify the whole range in
=O(n)= after you have bulk-modified the storage, instead of popping and pushing
every element.

** What is exposed, and what is not

=top()= returns a pointer to the winner, or =nullptr= when empty. =data()= and
=span()= expose the storage read-only, but in *heap order*, not sorted order, so
they are for feeding range algorithms, not for reading priorities in sequence.
There is deliberately no =begin= / =end= and no =operator==/<=>=: heap-layout
order is neither priority nor sorted order and depends on insertion history, so
iterating or comparing in it misleads. To consume in priority order, pop; to see
sorted order, copy from =span()= and sort the copy.

* Complexity

| operation                     | cost     |
|-------------------------------+----------|
| push / emplace                | O(log n) |
| pop                           | O(log n) |
| top                           | O(1)     |
| construct from list / range   | O(n)     |
| rebuild (re-heapify)          | O(n)     |
| data / span (read-only)       | O(1)     |
| clear / swap                  | O(1)     |

Constructing from a list or range is one =O(n)= =make_heap=, strictly cheaper
than =n= separate =O(log n)= pushes (=O(n)= versus =O(n log n)=). Memory
overhead is just the backing =std::vector='s geometric spare capacity.

* When to use it

Reach for =heap= when you repeatedly need the single most important element next
and the set keeps changing: event schedulers (min-heap, earliest event next),
A* / Dijkstra frontiers, top-K selection, bandwidth or job priority queues. The
contiguous array layout sifts one element along a single root-to-leaf path per
push or pop, with no node allocation. It is the right structure only when you
consume from *one* end (the top); if you also need the *worst* element cheaply,
you want a different structure (a balanced tree or a double-ended heap), since
this heap leaves everything below the root only partially ordered.

Choosing between the obvious candidates:

- versus =std::priority_queue=: this heap is the same algorithm on the same
  contiguous storage, but adds two things =std::priority_queue= hides. It returns
  =result<T>= from =pop= instead of leaving an empty-container pop as undefined
  behaviour, and it exposes the backing store read-only through =data()= /
  =span()= for feeding range algorithms (a bulk scan, a parallel reduction, a
  serialization pass) without draining the heap. Reach for =std::priority_queue=
  only when you want the exact standard-library type at an interface boundary.
- versus [[file:deque.org][deque]]: reach for =deque= when you want first-in or
  last-in order rather than highest-priority; a queue or stack is cheaper than a
  heap when priority is just arrival order.
- versus [[file:bag.org][bag]]: reach for =bag= when order is irrelevant and you
  just add, remove, and iterate; its =O(1)= swap-pop erase beats =O(log n)=
  sifting when you never ask "which is most important".

Note the read-only span and the absence of iteration. There is deliberately no
=begin= / =end= and no =operator== / =<=>=: heap-layout order is neither priority
order nor sorted order and depends on insertion history, so iterating or
comparing in it would mislead. =data()= and =span()= are =T const*= and
=std::span<T const>=, read-only on purpose, because writing through them could
break the invariant with no way to detect it. To consume in priority order,
=pop=; to read sorted order, copy from =span()= and sort the copy; to compare two
heaps, compare those projections, not the raw layout.

Invalidation and stability:

- A push or emplace that grows the backing vector invalidates =top()=,
  =data()=, =span()=, and any held pointer. =reserve= up front to keep them
  stable across a known number of pushes.
- *Any* mutation (=push=, =pop=, =emplace=, =rebuild=) can re-order elements, so
  treat =top()=, =data()=, and =span()= results as valid only until the next
  mutation, even when no reallocation occurs.

* Usage

#+BEGIN_SRC cpp
nexenne::container::heap<event, std::greater<event>> schedule;  // min-heap

schedule.push(e);                       // O(log n)
schedule.emplace(time, payload);        // construct in place

while (!schedule.empty()) {
  auto const next{schedule.pop()};      // result<event>; earliest first
  dispatch(*next);
}
#+END_SRC
