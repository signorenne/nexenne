#+TITLE: deque
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::deque<T>= is a double-ended queue backed by a single
contiguous, power-of-two buffer addressed as a ring. Pushing and popping at
either end is amortised =O(1)=, and indexed access is =O(1)= and cache-friendly.
When the buffer fills it doubles and re-packs, so the capacity is not fixed.

* The idea

A double-ended queue wants cheap insert and remove at both ends. =std::deque=
gets that from a map of fixed segments, which costs an extra indirection on every
access and scatters the data. A plain ring buffer is contiguous and index-cheap
but has a fixed capacity. =deque= takes the contiguous ring and lets it grow:
one allocation, direct addressing, and amortised growth, trading the segmented
layout's stable addresses for speed (a push that grows invalidates references).

* How it works

** Storage and addressing

The deque holds one buffer of =cap= slots (=cap= is a power of two), the index of
the front element, and a size. Logical index =i= maps to a physical slot by
adding the head and masking, so the ring wraps with a single =&= rather than a
modulo.

#+BEGIN_EXAMPLE
after push_back(1), push_back(2), push_front(0):   cap = 8

  index:    0    1    2    3    4    5    6    7
          +----+----+----+----+----+----+----+----+
          | 1  | 2  |  . |  . |  . |  . |  . | 0  |
          +----+----+----+----+----+----+----+----+
                                                ^ head = 7

  front-to-back order is d[0]=0, d[1]=1, d[2]=2
  slot_of(i) = (head + i) & (cap - 1)
#+END_EXAMPLE

A back push writes =slot_of(size)= and bumps the size; a front push moves the
head back one slot (with wrap, =(head + cap - 1) & (cap - 1)=) and writes there.
Both pops destroy the end element and step the size or head inward. Because head
and size are independent, the front can sit anywhere in the buffer and the live
elements can straddle the wrap point, as element =0= does in the diagram above
(it lives in slot 7 while elements 1 and 2 live in slots 0 and 1). Nothing is
ever moved on a push or pop that fits: the slot is addressed in place, which is
what keeps references stable until a growth.

** Growth

A push checks =size >= cap= first; when the buffer is full it calls =grow=, which
rounds the requested capacity up to a power of two (so the mask trick keeps
working), then re-packs. Walk a back push into a full =cap = 4= ring whose head
has wrapped:

#+BEGIN_EXAMPLE
full ring, cap = 4, head = 2, size = 4        new buffer, cap = 8

  index:  0    1    2    3                     index:  0   1   2   3  4..7
        +----+----+----+----+                        +---+---+---+---+----
        | C  | D  | A  | B  |                        | A | B | C | D |  .
        +----+----+----+----+                        +---+---+---+---+----
                  ^ head = 2                          ^ head = 0

  logical order A B C D (slots 2,3,0,1)        copied in logical order,
                                               so slot i now holds element i
#+END_EXAMPLE

=grow= allocates a buffer of =new_cap= slots, then loops =i= from 0 to =size=
and moves =m_data[slot_of(i)]= into =new_data[i]= (=construct_at= then
=destroy_at= the source), so the wrapped run is unrolled into a flat run with the
front landing at index 0. It frees the old buffer, sets =head = 0=, and the
original push then proceeds into the now-contiguous tail. =grow= is a no-op when
the buffer already fits (=new_cap <= cap=), which is what lets =reserve= and the
sizing constructor share it. Growth is the only =O(n)= event and is amortised
away over the pushes between growths (capacity at least doubles each time:
=grown_capacity= is eight on the first growth, then =cap * 2=). Storage comes
from the over-aligned =::operator new=, so an over-aligned =T= is placed
correctly, and the matching aligned =::operator delete= frees it.

** Values returned, errors, and copies

=front()= and =back()= return a pointer that is =nullptr= when the deque is
empty; =operator[]= is unchecked. =pop_front()= and =pop_back()= return the
removed element as =result<T>=, or =container_error::empty= when there is
nothing to remove. =push_*= and =emplace_*= are =noexcept= and terminate only on
allocation failure. The deque is copyable when =T= is, via a deep copy that
re-packs from the front, and always movable in =O(1)= by stealing the buffer.

* Complexity

| operation                       | cost           |
|---------------------------------+----------------|
| push_back / push_front          | O(1) amortised |
| emplace_back / emplace_front    | O(1) amortised |
| pop_back / pop_front            | O(1)           |
| operator[], front, back         | O(1)           |
| grow (buffer full)              | O(n) amortised |
| copy construct (deep, re-packs) | O(n)           |
| move construct / swap           | O(1)           |

Memory overhead is the spare ring capacity: =cap= is always a power of two and
at least double the size after a growth, so up to half the buffer can be unused
slots. A growth allocates a buffer twice the size (eight on the first), moves
every element into it, and frees the old one.

* When to use it

Reach for =deque= when you add and remove at *both* ends and want contiguous,
=O(1)= random access: work queues, sliding windows that grow and shrink at the
ends, breadth-first frontiers, double-ended caches, or a ring-style buffer that
may need to outgrow its initial size. The single power-of-two block gives faster
random access than =std::deque='s segmented map (one =&= mask, no per-access
indirection) and one allocation instead of a map of segment blocks.

Choosing between the obvious candidates:

- versus =std::deque=: reach for this one when random access and cache locality
  dominate and you can tolerate growth invalidating references; reach for
  =std::deque= when you need *stable* element addresses across pushes (it never
  relocates an element) or =O(1)= insert anywhere, not just at the ends.
- versus [[file:ring_buffer.org][ring_buffer]]: reach for =ring_buffer= when the
  capacity is genuinely fixed (a bounded log, a fixed-lag window) and you want
  the simplest possible storage with no reallocation; reach for =deque= when the
  bound is soft and a push must grow rather than overwrite or be rejected.
- versus [[file:bag.org][bag]]: reach for =bag= when order does not matter and
  you only add, remove, and iterate; its =O(1)= swap-pop erase is cheaper than
  maintaining end order.
- versus [[file:heap.org][heap]]: reach for =heap= when you want the
  highest-priority element next rather than the oldest or newest.

Invalidation and stability:

- A push or emplace that grows the buffer reallocates and invalidates *all*
  references, pointers, and =front= / =back= results. Construct with an
  =initial_cap= or call =reserve= up front to keep them stable across a known
  number of pushes.
- Without a growth, references stay valid across pushes and pops at the
  ends (no element is relocated, slots are addressed in place).
- =pop_front= / =pop_back= invalidate only the popped end's reference; the
  others remain valid.
- =operator[]= is unchecked, =front= / =back= return =nullptr= when empty.

* Usage

#+BEGIN_SRC cpp
nexenne::container::deque<task> work;

work.push_back(job);                 // enqueue at the back
work.emplace_front(urgent, args);    // jump the queue at the front

while (!work.empty()) {
  auto const next{work.pop_front()}; // result<task>; empty() guards it here
  run(*next);
}
#+END_SRC
