#+TITLE: gap_buffer
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::gap_buffer<T>= is the classic text-editor buffer: a logical
sequence stored in one backing vector with a movable gap of unused slots at the
cursor. Inserting and erasing at the cursor are =O(1)= because they only resize
the gap; random access stays =O(1)=; the cost is an =O(distance)= shift when the
cursor moves.

* The idea

Editing is local: most keystrokes land near where the last one did. A plain
=std::vector= makes an insert in the middle =O(n)= because every following
element shifts. A gap buffer keeps the shift but pays it only when the cursor
*moves*, not on every edit: it parks a block of free slots (the gap) right at the
cursor so an insert just drops into the next free slot and a delete just widens
the gap. Move the cursor once per edit session and a burst of edits there is each
=O(1)=.

* How it works

** Layout

The backing vector is read as three regions: the elements before the cursor, the
gap, and the elements after. A logical index maps to a physical one by skipping
the gap when it is at or past =gap_begin=.

#+BEGIN_EXAMPLE
gap_buffer<char> holding "hello", cursor after "he":

  pre        gap                  post
  +---+---+  +---+---+---+---+  +---+---+---+
  | h | e |  | . | . | . | . |  | l | l | o |
  +---+---+  +---+---+---+---+  +---+---+---+
             ^                  ^
        gap_begin (cursor)      gap_end

  logical order: h e | l l o     (| marks the cursor)
  physical(i) = i < gap_begin ? i : i + gap_size
#+END_EXAMPLE

** Editing at the cursor

=insert= drops a value into =gap_begin= and advances it (the gap shrinks by one);
=emplace= constructs the value first. =erase_forward= (the delete key) widens the
gap to the right over the following element, and =erase_backward= (backspace)
widens it to the left. All four are =O(1)=. Erase is a *logical* erase: it widens
the gap over the element rather than destroying it, so the element's destructor
runs when the slot is later reused by an insert or when the buffer is destroyed,
not at the erase call. =shrink_to_fit= closes the gap and reclaims the storage.

** Moving the cursor

=move_cursor_to= / =move_cursor_by= reposition the gap by shifting elements
across it, =O(distance)=, returning =container_error::out_of_range= for a target
outside =[0, size()]= (=move_cursor_by= checks the signed target before
delegating, so a negative overshoot is caught too). Moving *left* peels elements
off the pre region and drops them onto the gap's right edge; moving *right* does
the reverse. Walk one step left from the "hello" buffer above (cursor at 2):

#+BEGIN_EXAMPLE
move_cursor_to(1): one element ('e') crosses the gap left-to-right

  before                          after
  | h | e |. . . .| l | l | o |   | h |. . . .| e | l | l | o |
        ^gb        ^ge                ^gb        ^ge
                                  step: decrement gap_begin and gap_end,
                                  buffer[gap_end] = move(buffer[gap_begin])
#+END_EXAMPLE

Each step does one move, so the gap "rolls" over the elements between the old and
new cursor while its width stays constant. This is why a far cursor jump is
=O(distance)=: you pay one move per element crossed, which is also why parking the
cursor once per edit session and bursting edits there is the intended pattern.

** Growing

When an insert finds the gap empty (=gap_size() == 0=), =grow_gap= enlarges it.
It computes the post-region count, picks a new gap width (the larger of the
request and =gap_size * 2 + initial_gap=), resizes the backing vector, then
relocates the post region to the new tail. The post elements move *back to front*
(=j= from =post_count= down to 1) because the destination overlaps and sits
strictly above the source, so copying high indices first avoids clobbering a
not-yet-moved element. =gap_end= is then reset to where the post region now
begins, which is what reopens the gap.

#+BEGIN_EXAMPLE
insert with gap empty: resize, then slide post to the tail (high index first)

  before  | h | e | l | l | o |              (gap_size 0)
  resize  | h | e | l | l | o | . . . . . .  (vector grown)
  slide   | h | e | . . . . . . | l | l | o | (post moved right, gap reopened)
            ^gb                  ^ge
#+END_EXAMPLE

The gap slots hold default-constructed elements (=resize= value-initializes the
new tail and the vacated slots keep their moved-from objects), which is why =T=
must be default constructible and why an insert *assigns* into =gap_begin= rather
than constructing there.

** Iteration and comparison

=begin= / =end= give random-access iterators over the *logical* sequence, so
iteration, =operator[]=, =front= / =back=, and the reverse iterators all read in
order regardless of where the gap sits. =operator=== and =operator<=>= compare
the logical sequences, so two buffers with the same contents but different cursor
positions compare equal.

* Complexity

| operation                       | cost           |
|---------------------------------+----------------|
| insert / emplace at the cursor  | O(1) amortised |
| erase_forward / erase_backward  | O(1)           |
| move_cursor_to / move_cursor_by | O(distance)    |
| operator[], at, front, back     | O(1)           |
| iterate begin .. end            | O(n)           |
| operator== / operator<=>        | O(n)           |
| grow (gap exhausted)            | O(n) amortised |

Memory overhead is the gap: between =initial_gap= (16) and the high-water gap
size worth of unused slots, each a default-constructed =T=. =shrink_to_fit=
closes the gap and returns that storage.

* When to use it

Reach for =gap_buffer= when edits cluster around one position that moves slowly:
text and code editors, REPL input lines, command history, undo/redo timelines,
single-cursor log or transcript views. The pattern it rewards is "move the cursor
rarely, edit at it often"; the pattern it punishes is scattered random edits.

Choosing between the obvious candidates:

- versus =std::vector=: reach for =gap_buffer= when inserts and erases happen
  away from the end and cluster near one point; it turns =std::vector='s =O(n)=
  middle insert into =O(1)= at the cursor (paying the shift once on a cursor
  move, not on every edit). Reach for =std::vector= for append-only or
  read-mostly data, where its zero overhead and no gap-shift logic win outright.
- versus a rope (a balanced tree of chunks): reach for =gap_buffer= when the
  edited region is small enough that =O(distance)= cursor moves are cheap and you
  want one flat allocation, simple code, and =O(1)= random access. Reach for a
  rope when the sequence is large *and* edits scatter, so that a gap buffer would
  thrash on =O(distance)= jumps; a rope keeps both far edits and splits/joins
  near-=O(log n)= at the cost of pointer chasing and a heavier implementation.
- versus the siblings: for append-only or both-ends work prefer
  [[file:deque.org][deque]]; for a fixed-capacity ring prefer
  [[file:ring_buffer.org][ring_buffer]].

Invalidation and stability gotchas:

- A cursor move (=move_cursor_to= / =move_cursor_by=) physically shifts elements
  across the gap, so it invalidates references, pointers, and iterators to the
  elements it crossed; treat the whole logical sequence as relocated.
- A growth reallocates the backing vector, invalidating *all* references,
  pointers, and iterators. =reserve= up front to keep them stable across a known
  number of inserts.
- Even a plain =insert= or =erase= at the cursor shifts where the gap sits, so
  read an =operator[]= or =at= result before the next edit and never hold an
  iterator across one.
- Erase is *logical*: =erase_forward= / =erase_backward= widen the gap over the
  element rather than destroying it, so its destructor runs only when the slot is
  reused by a later insert or when the buffer is destroyed. A =gap_buffer= of
  resource-holding types therefore releases those resources lazily, which matters
  for file handles, locks, or large buffers.
- =operator[]= and =at= with an in-range or out-of-range index differ:
  =operator[]= is unchecked, while =at= returns =nullptr= past =size()=, as do
  =front= / =back= on an empty buffer.

* Usage

#+BEGIN_SRC cpp
nexenne::container::gap_buffer<char> line;

for (char const ch : "hello") { line.insert(ch); }   // O(1) each at the cursor
if (line.move_cursor_to(2).has_value()) {            // reposition once
  line.insert('!');                                  // O(1) burst of edits here
}
static_cast<void>(line.erase_backward());            // backspace, O(1)
#+END_SRC
