#+TITLE: union_find
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::union_find<Index>= is a disjoint-set ("union-find")
structure: it maintains a partition of the integer nodes =[0, n)= into disjoint
sets and answers "are these two in the same set" and "merge these two sets" in
near-constant amortised time. =Index= defaults to =std::uint32_t=, with
=union_find_u16= / =u32= / =u64= aliases.

* The idea

Tracking which things are grouped together, and merging groups as you learn of
new connections, is the recurring shape of connected-component problems. Rerunning
a graph traversal after every new edge is wasteful. A disjoint-set forest instead
represents each set as a tree whose root names the set, so "same set" is "same
root" and "merge" is "point one root at the other". Two cheap heuristics, union by
size and path compression, keep those trees almost flat, which is what drives the
cost down to effectively constant.

* How it works

** Forest of parent links

Each node stores a parent; a root points at itself. =find(i)= walks parents to
the root, and per-root sizes (kept in a second array) drive the merge choice.

#+BEGIN_EXAMPLE
union_find{5} after unite(0, 1) then unite(1, 2):  sets {0,1,2}, {3}, {4}

  parent:  [ 1, 1, 1, 3, 4 ]      each node points at its parent
  node:       0  1  2  3  4       a root points at itself

  as a forest:
        1            3      4
       / \
      0   2
#+END_EXAMPLE

** find with path halving, step by step

=find(i)= bounds-checks =i= (an out-of-range index returns
=container_error::out_of_range=), then walks =m_parent= until it reaches a node
that is its own parent, the root. On each step it does one extra write,
=m_parent[current] = m_parent[m_parent[current]]=, which re-points =current= at
its grandparent before advancing. That is path halving: every other node on the
walk is lifted one level, so the chain is roughly halved in length each pass and
later finds along it are shorter. Because of that write, =find= *mutates* even
though it reads like a query.

#+BEGIN_EXAMPLE
find(0) on the chain 0 -> 1 -> 2 -> 3 (3 is the root):

  before:        3            walk: 0, then 1, then 2, reach root 3
                 |
                 2
                 |
                 1
                 |
                 0

  during: at node 0, point 0 at its grandparent 2; advance to 2.
          at node 2, 2's parent is 3 (the root); advance, loop ends.

  after:         3            0 was halved up to 2; the chain is shorter
                / \
               1   2
                   |
                   0
#+END_EXAMPLE

=root_of(i)= is the =const= alternative: it walks the same parent chain but
performs no writes, so it does no flattening and is =O(h)= for chain length =h=,
but it is usable from a constant context or a comparison that must not mutate
(it is what =same_partition= uses). Use =find= when you want the side benefit of
compression, =root_of= when you must stay =const=.

** unite by size, step by step

=unite(a, b)= calls =find= on both (so both chains get compressed) and, if the
roots already match, returns =false= with no change. Otherwise it compares the
two roots' sizes, swaps so =big_root= is the larger (a tie leaves =b='s root as
=big_root=, so the second argument wins), points =m_parent[small_root] = big_root=
to hang the smaller tree under the larger, folds the small set's size into
=m_set_size[big_root]=, zeroes the small root's size entry (a non-root no longer
carries a size), and drops =m_set_count= by one. Attaching the *smaller* tree is
what bounds the height: a node's depth only grows when its set is the smaller
party in a merge, which can happen at most =log2(n)= times, so trees stay
=O(log n)= tall even before any compression.

** Why the two heuristics together are near-constant

Union by size alone caps tree height at =O(log n)=, giving =O(log n)= finds. Path
halving alone also helps but can still see deep chains. Used together they
amortise to =O(alpha(n))= per operation, where =alpha= is the inverse Ackermann
function: union by size keeps trees shallow to begin with, and every =find=
permanently flattens the part of the chain it touches, so repeated queries pay
the depth cost only once. =connected(a, b)= is =find(a) == find(b)= (both
finds compress), and =size_of(i)= returns =m_set_size= at =i='s root.

* Complexity

| operation                              | cost                   |
|----------------------------------------+------------------------|
| find / unite / connected / size_of     | amortised O(alpha(n))  |
| make_set                               | amortised O(1)         |
| root_of (no compression)               | O(h), chain length h   |
| reset                                  | O(n)                   |
| clear / swap / parents / set_sizes     | O(1)                   |
| same_partition                         | O(n * alpha(n))        |

=alpha(n)= is the inverse Ackermann function, below 5 for any =n= that fits in
memory, so the per-operation cost is effectively constant. Memory is two vectors
of =n= entries (parent links and per-root sizes).

* When to use it

Reach for =union_find= when you incrementally learn that things are connected and
need fast grouping queries: connected-component and island detection (the solver
islands of a physics step), Kruskal-style minimum spanning trees, and
equivalence-class tracking while building a graph.

*Versus re-running a graph traversal:* a BFS or DFS answers "are x and y
connected" in =O(V + E)= per query and must be redone after every new edge.
=union_find= folds each edge in at =O(alpha(n))= and answers a query at the same
cost, so for the streaming "add edges, ask connectivity" pattern it is
dramatically faster and needs no adjacency lists, just two integer arrays. Prefer
a real traversal when you need the *paths* themselves, per-edge structure, or
edge removal: =union_find= tracks only which set each node lands in, not how it
got there.

Things to keep in mind:

- *It only merges.* Sets join and never split; there is no "disconnect" and no way
  to undo a =unite=. To shrink or rebuild a partition, =reset(n)= back to
  singletons or =clear=, then replay the edges you still want. Model node removal
  by rebuilding from scratch.
- *find mutates.* Path compression rewrites parent links, so =find=, =connected=,
  =size_of=, and =unite= are all non-const; do not call them on a =const= or
  shared structure expecting it to stay put. Use =root_of= for a =const= query,
  accepting its =O(h)= cost and that it leaves the tree uncompressed.
- *Indices are raw integers in =[0, size())=.* There is no string or pointer
  keying; map your objects to dense indices yourself (=make_set= hands out the
  next one). =find=, =connected=, =size_of=, and =unite= return
  =container_error::out_of_range= on a bad index rather than reading out of
  bounds, while =root_of= assumes a valid index as a precondition and does not
  check.
- *Comparison is explicit.* There is no =operator===; =same_partition= compares
  two structures by classification in =O(n * alpha(n))= regardless of compression
  history (it uses =root_of=, so it never mutates), and is given a visible name so
  that cost never hides behind a generic comparison in a hot loop. The =parents()=
  and =set_sizes()= spans are read-only and invalidated by any mutating call,
  including a bare =find=.

* Usage

#+BEGIN_SRC cpp
nexenne::container::union_find_u32 uf{node_count};

for (auto const [a, b] : edges) {
  static_cast<void>(uf.unite(a, b));   // O(alpha(n))
}
auto const components{uf.count()};     // number of disjoint sets
if (*uf.connected(x, y)) { ... }       // result<bool>, O(alpha(n))
#+END_SRC
