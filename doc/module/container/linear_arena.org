#+TITLE: linear_arena
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::linear_arena<N>= is a bump allocator over an inline =N=-byte
buffer. =allocate= hands out a pointer into the buffer in =O(1)= (a pointer bump
plus a compare), there is no per-allocation free, and =reset= releases
everything at once, also in =O(1)=. It never touches the heap.

* The idea

A lot of allocation is phase-scoped: everything allocated during a frame, a
request, or a parse is thrown away together at the end. A general allocator pays
for per-object bookkeeping and freeing that such code never uses. A bump
allocator drops all of it: allocation is just "advance a cursor," and freeing is
"move the cursor back to the start." =linear_arena= puts the buffer inline (no
heap, stack-bufferable) and hands out raw memory, leaving object lifetime to the
caller.

* How it works

** Memory layout

The arena is an aligned inline byte buffer plus a single =offset= cursor (and a
high-water mark for diagnostics). =allocate= rounds the cursor up to the
requested alignment, checks the block fits, advances the cursor, and returns the
old position.

#+BEGIN_EXAMPLE
linear_arena<64> after allocate(8,8), allocate(4,4):

  +----------------------------------------------------------+
  | [ 8-byte block ][ 4 ][ . . . . . free . . . . . . . . ]  |
  +----------------------------------------------------------+
  ^                      ^
  buffer start           offset = 12   (bytes_used)

  reset() moves offset back to 0; rewind_to(k) moves it back to k.
#+END_EXAMPLE

** Allocation and lifetime

=allocate(size, alignment)= returns raw =void*=; the typed =allocate<T>(count)=
sizes and aligns for =T=; =emplace<T>(args...)= also constructs the object.
Allocation fails with =container_error::full= when the block does not fit. The
arena hands out *raw memory* and does not track or run destructors, so for a
non-trivially-destructible =T= the caller must =std::destroy_at= the object
before the storage is reset. Trivial types can simply be abandoned.

** Releasing

There are three ways to give memory back, all =O(1)= and none of which run
destructors: =reset()= drops everything; =rewind_to(offset)= drops back to a
saved =bytes_used()= (the basis for [[file:scratch_pad.org][scratch_pad]]'s RAII checkpoints); and
letting the arena go out of scope drops its whole buffer. =high_water_mark()=
records the peak =bytes_used()=, which is handy for sizing the buffer
empirically.

* Complexity

| operation                          | cost |
|------------------------------------+------|
| allocate(size, alignment)          | O(1) |
| allocate<T>(count) / emplace<T>    | O(1) |
| reset (drop everything)            | O(1) |
| rewind_to(offset)                  | O(1) |
| bytes_used / bytes_available       | O(1) |
| high_water_mark                    | O(1) |

Every operation is =O(1)=: a bump is a round-up, a compare, and a cursor
advance, and releasing is a single cursor assignment. Memory overhead is fixed
and inline, the =N=-byte buffer aligned to =max_align_t= plus two =size_type=
cursors (=offset= and the high-water mark), with no heap and no per-allocation
header. The cost of that simplicity is internal fragmentation: alignment padding
is dead space, and a workload that needs less than =N= at once still reserves
the whole buffer.

* When to use it

Reach for =linear_arena= when allocations are phase-scoped, everything made
during a frame, request, or parse is discarded together at the end. Bump
allocation is far cheaper than a general allocator there because it skips all the
free-list bookkeeping that such code never uses, and the inline buffer keeps it
off the heap (stack-bufferable, usable in freestanding code). Versus
=std::pmr::monotonic_buffer_resource= it is a direct, non-virtual call with no
type-erasure hop, at the cost of no allocator-aware-container adapter.

The model has two hard rules:

- *No individual free.* There is no =deallocate(ptr)=, you cannot give back one
  allocation. Memory comes back only in bulk: =reset()= drops everything,
  =rewind_to(saved)= drops back to an earlier =bytes_used()= (the basis for
  [[file:scratch_pad.org][scratch_pad]]'s RAII checkpoints), or the arena going
  out of scope drops its whole buffer. For scoped temporaries, prefer a
  [[file:scratch_pad.org][scratch_pad]] over a manual =rewind_to=.
- *Raw memory, no destructors.* The arena hands out untyped storage and never
  runs destructors. =reset= and =rewind_to= just move the cursor, so a
  non-trivially-destructible =T= built in arena storage must be =std::destroy_at=
  'd before the cursor passes it, or it leaks. A trivially destructible =T= can
  simply be abandoned.

Invalidation: =reset= invalidates *every* pointer into the arena; =rewind_to(k)=
invalidates every pointer at or above offset =k= and leaves those below it
valid. Allocation never relocates earlier blocks, so a pointer stays valid until
a =reset= or =rewind_to= passes it (or the arena is destroyed).

* Usage

#+BEGIN_SRC cpp
nexenne::container::linear_arena<4096> arena;

for (auto const& frame : frames) {
  auto const nodes{arena.allocate<node>(frame.count)};   // raw storage
  if (!nodes.has_value()) { handle(nodes.error()); }     // container_error::full
  // ... fill and use the scratch ...
  arena.reset();                                         // release the frame, O(1)
}
#+END_SRC
