#+TITLE: object_pool
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::object_pool<T, N>= is a fixed-capacity pool of =N=
pre-allocated, aligned slots for =T=. It hands out a free slot and reclaims one
in =O(1)=, with no heap and no allocator after construction. A free list recycles
the most-recently-freed slot first, keeping it hot in cache.

* The idea

When you spawn and retire many short-lived objects of one type, calling =new= and
=delete= per object pays the general allocator's cost over and over, and a bump
arena like [[file:linear_arena.org][linear_arena]] cannot free a single object, only the whole region at
once. A pool sits between them: reserve the storage once, then hand individual
slots out and take them back in constant time. The bookkeeping is a stack of
free-slot pointers, so acquisition and release are a single push or pop.

* How it works

** Slots and the free list

The pool owns two arrays inline: =N= raw, =T=-aligned slots (each a
=sizeof(T)=-sized byte buffer with =alignof(T)= alignment, so a =T= can be
constructed there) and a free list, a [[file:static_vector.org][static_vector]] of slot pointers. The free
list is the only moving part: it is a stack holding a pointer to every slot that
is not currently handed out. =size()= is simply =N= minus the free list's size,
so the pool itself stores no separate live count.

The constructor seeds the free list in reverse (slot =N-1= first, slot 0 last) so
the very first =acquire= pops slot 0 and the slots are handed out in ascending
order while the pool fills cold.

#+BEGIN_EXAMPLE
object_pool<T, 4>: four slots plus a stack of free-slot pointers

  slots:  [ s0 ][ s1 ][ s2 ][ s3 ]      raw aligned storage, fixed addresses
  free:   [ &s3, &s2, &s1, &s0 ]        back is the next one out

  acquire() pops &s0, hands out s0      free: [ &s3, &s2, &s1 ]
  release(p) pushes the slot back       the slot rejoins the free list
#+END_EXAMPLE

** Walking an acquire and a release

An =acquire()= is a single pop. If the free list is empty the pool is exhausted
and it returns =container_error::full=; otherwise it reads the back pointer,
=pop_back='s it off, bumps the high-water mark if the new live count exceeds it,
and returns =slot.ptr()=, a =T*= into raw, uninitialised storage.

A =release(p)= is a single push. The pool turns =p= back into a slot pointer
(the slot's storage sits at offset 0, so a =T*= into a slot *is* the slot's
address) and =push_back='s it onto the free list. It runs no destructor.

Because both ends operate on the back of the free list, recycling is LIFO: the
slot you just released is the next one acquired, so it is still hot in cache and
its TLB entry is still warm. The slot addresses never change for the life of the
pool, so a handle is a stable pointer.

#+BEGIN_EXAMPLE
acquire then release of the same slot: LIFO keeps it hot

  free: [ &s3, &s2, &s1 ]   p = acquire()  ->  s0 leaves (back was &s0 popped earlier)
  ...use p...
  free: [ &s3, &s2, &s1 ]   release(p)     ->  push &s0
  free: [ &s3, &s2, &s1, &s0 ]
  q = acquire()             ->  pops &s0 again: q == p, same warm slot
#+END_EXAMPLE

** Two API tiers: who runs the constructor and destructor

The two tiers differ only in who manages =T='s lifetime.

- *Raw tier:* =acquire()= returns raw =T*= (or =container_error::full=) and
  constructs *nothing*: you =std::construct_at= the =T= yourself. =release(p)=
  reclaims the slot and runs *no* destructor: you =std::destroy_at= first. Use
  this when you want to place an object only sometimes, or manage the lifetime by
  hand.
- *Object tier:* =emplace(args...)= calls =acquire()= then =std::construct_at=
  with your forwarded arguments in one step (constrained to
  =std::constructible_from<T, Args...>=); =destroy(p)= calls =std::destroy_at=
  then pushes the slot back. The pool runs both the constructor and the
  destructor for you.

The pool only ever constructs and destroys =T= *in place* and never moves or
copies it, so an immovable or non-copyable =T= (a mutex, an atomic, a type with
deleted move) works exactly like any other.

Both =release= and =destroy= validate =p=: =container_error::not_found= for a
null pointer, =container_error::out_of_range= for a pointer that does not fall
within this pool's slot array (a range check against the slots' address span).
They do not, and cannot cheaply, detect a double release of a still-in-range
slot.

** High-water mark

=high_water_mark()= records the largest =size()= seen since construction or the
last =clear_high_water_mark()=. =acquire= updates it on the way up; =release=
never lowers it. Run a representative workload, read the mark, and size =N= to it
instead of guessing.

* Complexity

| operation                           | cost |
|-------------------------------------+------|
| acquire / release                   | O(1) |
| emplace / destroy                   | O(1) |
| size / empty / full                 | O(1) |
| high_water_mark                     | O(1) |

Memory is =N= slots of =sizeof(T)= (each =alignof(T)=-aligned) plus the free
list of =N= pointers, all inline, allocated once at construction and never
again.

* When to use it

Reach for =object_pool= when many objects of one type are spawned and retired
individually and you want =O(1)= recycling with no per-object allocation:
particle systems, fixed rosters of reusable buffers or connections, per-frame
transient scratch objects, and embedded contexts where =new= is forbidden.
Because it constructs and destroys in place and never relocates, it also holds
*immovable* types (a mutex, an atomic) that a vector-backed container cannot.

The deciding question is *how you free*:

- Free objects *one at a time*, in any order, and reuse their storage: that is
  exactly =object_pool=. The free list pays one pointer per slot for the
  privilege.
- Free *all at once* and never individually: a bump [[file:linear_arena.org][linear_arena]] is leaner. Its
  bookkeeping is a single offset (no free list, no per-slot pointer), it packs
  mixed types and sizes back to back, but it can only rewind the whole region, it
  cannot reclaim one object.
- Need an *unbounded* or growable count, or a general-purpose lifetime: plain
  =new= / =delete= or =std::unique_ptr=. =object_pool= is a fixed =N= and reports
  =container_error::full= when exhausted; it never grows.

Gotchas:

- *release does not destroy.* =release= and the raw =acquire= tier skip the
  destructor entirely. An object left in a released or reused slot has its
  destructor *silently skipped* (a leaked socket, an unlocked-forever mutex)
  unless you =std::destroy_at= it first. Use the object tier (=destroy=) and this
  cannot happen.
- *No double-free or use-after-release guard.* Releasing the same in-range slot
  twice pushes it onto the free list twice, so a later =acquire= can hand the
  same slot to two callers. Reading or writing through a pointer after releasing
  it is a use-after-free against whoever acquired it next. The pointer validation
  only catches null and out-of-range, not these.
- *Non-copyable and non-movable.* The pool hands out interior pointers, so all
  copy and move operations are deleted. A handle from =acquire= or =emplace=
  stays valid until you release it, and the pool itself must outlive every
  handle.
- *Exhaustion is a value, not an exception.* Every operation is =noexcept=;
  =acquire= and =emplace= return =container_error::full= when no slot is free.
  Check =has_value()=.

* Usage

#+BEGIN_SRC cpp
nexenne::container::object_pool<connection, 64> pool;

auto const c{pool.emplace(host, port)};   // acquire a slot and construct
if (!c.has_value()) { handle(c.error()); } // container_error::full when exhausted
use(*c);
static_cast<void>(pool.destroy(*c));       // destroy and recycle, O(1)
#+END_SRC
