#+TITLE: bag
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=nexenne::container::bag<T>= is an unordered, contiguous multiset: a collection
where order is genuinely irrelevant and the operations that matter are add,
remove, and iterate. Removal is =O(1)= because it swaps the last element into the
gap and pops, rather than shifting to preserve order. Duplicates are allowed.

* The idea

A surprising amount of code keeps a "set of things currently active": live
entities this frame, pending jobs, dirty nodes. The order of that set carries no
meaning, and uniqueness often does not either, yet the usual tools fight you:
=std::vector::erase= shifts every following element to keep order you do not
care about, and =std::multiset= needs a comparator and scatters its nodes across
the heap. =bag= keeps the contiguous storage of a vector but trades order for an
=O(1)= erase, which is the right deal when order is not part of the contract.

* How it works

** Storage

A =bag= is just a =std::vector<T>= with a different contract, so it inherits
contiguous storage, cache-friendly iteration, and =data()= / =span()= for
=std::ranges= interop, and follows the rule of zero (its copy, move, and
destructor are the vector's).

** Swap-pop erase

The one real mechanism is swap-pop: to remove the element at index =i=, move the
last element into slot =i= and pop the back. That is =O(1)= and the reason order
is not preserved.

#+BEGIN_EXAMPLE
erase_at(1) on [ A B C D ]:

  [ A | B | C | D ]     move last (D) into slot 1, then pop the back
        ^           ^
        i=1         last
  [ A | D | C ]         B is gone; D took its place; size shrank by one
#+END_EXAMPLE

=erase_at(i)= is bounds-checked and returns =result<void>=
(=container_error::out_of_range= past the end). =erase_first(value)= finds the
first match and swap-pops it, returning whether it removed anything.
=erase_all(value)= walks from the back so every swap-pop stays =O(1)=, giving
=O(size)= overall, and returns how many it removed.

** What is deliberately absent

There is no mid-range =insert= (order is meaningless), and no =operator=== or
=operator<=>=: because erase reshuffles, a sequence comparison would reflect
operation history rather than logical contents. A caller that needs multiset
equality should compare sorted or hashed copies. =contains= is also omitted
because =std::ranges::contains= already works on the bag through its iterators.

* Complexity

| operation                  | cost            |
|----------------------------+-----------------|
| insert / emplace           | O(1) amortised  |
| erase_at (swap-pop)        | O(1)            |
| erase_first(value)         | O(size)         |
| erase_all(value)           | O(size)         |
| operator[], data, span     | O(1)            |
| iterate begin .. end       | O(n)            |
| clear / swap               | O(1) / O(1)     |

=erase_first= and =erase_all= are =O(size)= because they scan for the match; the
removal itself is still =O(1)= swap-pop. Memory overhead is just the backing
=std::vector=: contiguous storage with the usual geometric spare capacity (use
=reserve= to size it and =shrink_to_fit= to reclaim it).

* When to use it

Reach for =bag= when the collection is a set of currently-active things whose
order carries no meaning, entities live this frame, pending jobs, dirty nodes,
and the hot path is add, remove, and iterate. The swap-pop erase is the win: it
is =O(1)= where a =std::vector::erase= would shift every following element to
preserve an order you do not need.

Prefer a sibling when:

- order matters at either end, a work queue or sliding window, use
  [[file:deque.org][deque]], whose pushes and pops keep front-to-back order.
- you need the highest-priority element out next, not any element, use
  [[file:heap.org][heap]].
- you genuinely need ordered or stable-positioned storage; reach for a plain
  =std::vector= and pay its =O(n)= ordered erase.

Invalidation and stability:

- Erase does *not* preserve order: =erase_at=, =erase_first=, and =erase_all=
  move the former last element into the freed slot, so indices other than the
  removed one can shift, never hold an index across an erase.
- A push or emplace that grows the backing vector reallocates and invalidates
  *all* references, pointers, and iterators (and the =data()= / =span()=
  result). =reserve= up front to keep them stable across a known number of
  inserts.
- =clear= keeps capacity (existing capacity-based pointers stay valid as
  storage, the elements are gone); =shrink_to_fit= may reallocate and
  invalidate everything.

* Usage

#+BEGIN_SRC cpp
nexenne::container::bag<entity_id> active;

active.insert(id);                 // O(1), order not preserved
active.erase_first(dead_id);       // O(1) swap-pop, returns bool
active.erase_all(stale_id);        // remove every match, returns the count

for (auto const id : active) {     // contiguous, cache-friendly
  update(id);
}
#+END_SRC
