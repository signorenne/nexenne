#+TITLE: trigonometry at three speeds
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

Sine, cosine, tangent and their inverses, offered at three deliberately distinct
cost/accuracy points so the caller chooses: full-precision libm (=sincos=,
plus =sin= / =cos= / =tan= in [[file:angle.org][angle]]), =constexpr= polynomial approximations
(=fast_sin=, =fast_cos=, =fast_sincos=, =fast_tan=, =fast_asin=, =fast_acos=,
=fast_atan=, =fast_atan2=), and a runtime lookup table (=lut_sin=, =lut_cos=).
Plus the angle-aware =angle_diff= and =lerp_angle= that always take the short way
around.

* The idea

Not every sine needs all of libm's precision, and not every target has a fast
libm. So instead of one =sin=, the header exposes a *menu* with honest cost
labels: the default is full precision; =fast_*= trades a few digits for a
branch-light =constexpr= polynomial; the LUT is there for constrained targets,
with the warning that on a modern desktop it is usually *slower* than libm
(memory latency beats arithmetic). The unifying trick on the fast path is *range
reduction*: fold any angle into =[-pi/4, pi/4]= where a short polynomial is
accurate, then fix up the sign and the sin/cos swap by quadrant.

* How it works

** sincos and the pair

=sincos= returns a =sin_cos=, a sine and cosine of the same angle returned
together. Computing both at once is the common case (rotations, polar to
cartesian) and is cheaper shared: one range reduction, and libm fuses
=std::sin=+=std::cos= into a single hardware sincos. See the type's own comment
for the full rationale and the optimizer caveat.

** The polynomial fast path

=poly_sincos= reduces =x= by =k = round(x / (pi/2))=, evaluates a degree-7 sine
and degree-8 cosine Horner polynomial on the reduced argument, then selects the
output by =k mod 4=:

#+BEGIN_EXAMPLE
quadrant k mod 4:   0 -> ( s,  c)   1 -> ( c, -s)
                    2 -> (-s, -c)   3 -> (-c,  s)
#+END_EXAMPLE

So =fast_sin= and =fast_cos= both need the reduced sine *and* cosine (the quadrant
decides which becomes the answer), which is exactly why the pair form costs no
more than either single. Accuracy is around 3e-7 near the origin and degrades by
roughly a decimal per period as the range reduction loses low bits to
floating-point cancellation in =reduced - k*(pi/2)= (=k= itself stays exact), so
the fast path is for bounded angles (wrap first with [[file:angle.org][wrap_signed]] if needed). The
inverse trig (=fast_asin= via Abramowitz and Stegun, =fast_atan= via a degree-11
fit with the =atan(1/x)= fold) follows the same approximate-but-constexpr spirit.

** The lookup table

=lut_sin= / =lut_cos= read a compile-time-generated 1024-entry sine table
(=inline constexpr=, so it lives in read-only data with no runtime init) and
linearly interpolate; cosine reuses the sine table at a quarter-turn offset.
Because =std::sin= is not =constexpr=, the table is seeded at compile time from
the polynomial approximation (not exact sines), which sets the accuracy floor.
Accuracy around 4.7e-6. Useful on a target with slow or absent libm; measure before
preferring it on a desktop.

** Short-way angle math

=angle_diff(a, b)= is =a - b= wrapped to =[-pi, pi)=, the shortest signed gap even
across the +/-pi seam. =lerp_angle= interpolates along that shortest arc, so
blending a heading from 350 to 10 degrees passes through 0, not all the way back
through 180.

* When to use which

- Default to =sincos= / =sin= / =cos=: full precision, and on modern hardware as
  fast as anything once range reduction is counted.
- =fast_*= for bounded angles in a hot loop where a few ULP do not matter and you
  want a =constexpr= result.
- =lut_*= only on a constrained target, or after measuring a win.
- =angle_diff= / =lerp_angle= whenever an angle may have wrapped and you need the
  short way around.

* Usage

#+BEGIN_SRC cpp
namespace nm = nexenne::math;

auto const sc{nm::sincos(nm::radians_d{theta})};   // both at once, full precision
auto const m00{sc.cos()}, m01{-sc.sin()};          // build a 2D rotation

constexpr auto s{nm::fast_sin(nm::radians_d{0.3})};  // approximate, constexpr
auto const blend{nm::lerp_angle(a, b, 0.5)};         // short way around
#+END_SRC
