#+TITLE: rate limiter
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

=rate_limiter<Clock>= is a token-bucket rate limiter with lazy refill: it caps how
often something may happen (log lines, retries, requests) while allowing a bounded
burst. No thread, no allocation, just arithmetic on each call.

* The idea

A token bucket holds up to =capacity= tokens and gains =refill_per_sec= tokens per
second. An action costs a token; if the bucket has one, it proceeds, otherwise it
is throttled. Starting full lets a short burst through immediately, then the
refill rate paces the steady state. "Lazy" means the bucket does not refill on a
timer, it computes how many tokens it *would* have accrued since it was last
touched, the moment you ask, so it needs no background work.

* How it works

The limiter stores the token count, the capacity and rate, and the instant it was
last refilled. =try_acquire(n)= first does a lazy =refill()= (add =elapsed_seconds
* rate= tokens, capped at capacity), then consumes =n= if at least =n= are
available, returning whether it did. Tokens are =double=, so fractional rates (one
token every 1.5 s) are exact. Several robustness details:

- The first touch only *anchors* the clock (no spurious initial refill).
- A backward clock step (only possible by abusing =manual_clock=; a steady clock
  cannot) is clamped to zero elapsed, so the bucket never loses tokens to it.
- A small epsilon absorbs the few ULPs of float drift in the refill arithmetic, so
  an acquire that is mathematically exactly affordable is not rejected; the count
  is clamped non-negative afterward so the epsilon cannot accrue into real debt.
- A NaN or negative =n= is rejected; a zero =n= trivially succeeds.

=tokens()= reports the available count after a refill. =until_next_token(n)=
returns how long to wait for =n= tokens (zero if already available,
=duration::max()= if the rate is zero and the bucket is short, otherwise the
rounded-*up* wait so a caller that sleeps then retries does not wake one ULP
early). =reset()= refills to full, =drain()= empties.

#+BEGIN_EXAMPLE
rate_limiter{3, 10}        // capacity 3, 10 tokens/sec (one per 100 ms)
try_acquire x5  -> true, true, true, false, false    // burst of 3, then empty
+100 ms, try_acquire -> true     // one token refilled
       try_acquire -> false      // and spent
#+END_EXAMPLE

* Complexity

| operation                       | cost |
|---------------------------------+------|
| try_acquire / tokens / until_next_token | O(1) (one now() read) |

It never allocates and reads the clock once per call.

* When to use it

Reach for =rate_limiter= to throttle or pace: drop log messages once a budget is
spent, space out retry attempts, apply back-pressure to a producer, cap an event
emission rate. Pair =until_next_token= with a sleep or an event-loop timer when you
want to block until the next token rather than poll-and-drop.

Prefer a sibling when:

- you want a *fixed cadence* rather than a burstable budget: [[file:polling-timers.org][interval]] fires every
  period exactly.
- you want a *one-shot or periodic callback*: [[file:polling-timers.org][alarm]].

Things to keep in mind:

- The bucket starts *full*, so the first burst can be up to =capacity= before
  pacing kicks in; construct and =drain()= if you want to start empty.
- Tokens are floating point; the epsilon tolerance means "affordable" is within a
  few ULPs, not bit-exact.

* Usage

#+BEGIN_SRC cpp
nexenne::chrono::rate_limiter<> log_rl{10.0, 2.0};   // burst 10, then 2/sec
if (log_rl.try_acquire()) {
    emit_log(line);                                   // else drop it
}
#+END_SRC
