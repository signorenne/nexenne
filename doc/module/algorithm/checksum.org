#+TITLE: checksum
#+OPTIONS: toc:nil num:nil ^:nil
#+STARTUP: showall

* Purpose

Integrity checks for detecting accidental corruption: a fast lightweight family
(=modular_sum=, the Adler and Fletcher checksums) and a configurable,
high-detection one (=crc=). All take a byte span or a string view; none are
cryptographically secure, so they guard against transmission and storage errors,
not tampering. Each is a generic engine parameterised by a spec, exactly the
same shape: =crc_spec= for CRC, =modular_sum_spec= for the sum family.

- =modular_sum= : two running sums modulo =M=, parameterised by the input unit
  size, sum width, and modulus. Named presets: =adler32= (zlib/PNG, prime
  modulus 65521), and Fletcher =fletcher16= / =fletcher32= / =fletcher64=. A
  \p seed continues a checksum across chunks. Cheap, but weaker than CRC on
  short messages.
- =crc= : a generic table-driven CRC engine parameterised by a =crc_spec=
  (the Rocksoft model), with named presets for the common CRC-8, CRC-16, and
  CRC-32 variants, =crc8= / =crc32= / =crc32c= convenience wrappers, and a
  streaming =crc_ctx=. The whole engine is =constexpr=.

* How it works

** modular_sum (Adler and Fletcher)

It keeps two sums modulo =M=: =sum1= adds each input unit, =sum2= adds each
=sum1=, and the result is =(sum2 << width) | sum1=. The members differ only in
three knobs the spec captures: the unit size (1 byte for Adler-32 and
Fletcher-16, a 16- or 32-bit little-endian word for Fletcher-32/64), the sum
width, and the modulus. Adler-32 uses the prime 65521, which detects slightly
more than the composite moduli (2^8-1, 2^16-1, 2^32-1) the Fletcher members use.
Taking the modulus on every unit is wasteful, so the engine defers it: it sums
up to 5552 units before reducing, which (with a 64-bit accumulator) cannot
overflow for any member and gives a result identical to the per-unit version at
a fraction of the divisions. Because the two sums fully describe the state, an
interrupted checksum resumes by passing the previous result back as the =seed=.

** CRC

A CRC treats the message as the coefficients of a polynomial and returns the
remainder after dividing by a fixed generator polynomial; that remainder catches
all single-bit errors, all burst errors up to the register width, and most
others, which is why CRCs dominate wire and storage formats. Six parameters (the
Rocksoft model) pin down a specific CRC: the register =width=, the =poly=, the
=init= value, whether input bytes and the output register are bit-reflected
(=ref_in= / =ref_out=), and the final =xor_out=. The catalogue at
https://reveng.sourceforge.io/crc-catalogue/all.htm lists about a hundred named
CRCs; each is one =crc_spec= literal.

Rather than divide bit by bit, the engine builds a 256-entry table once per spec
(processing a whole byte per step) at =constexpr= time, so it lives in ROM and
costs only a load and two XORs per byte at run time. =crc_ctx= exposes the same
computation incrementally: =update= folds in each chunk, =value= applies output
reflection and the final XOR to a copy of the register without disturbing it.

* When to use it

- A quick running check over non-adversarial data, or resuming a check across
  chunks cheaply: =adler32= (or a Fletcher preset to match an existing format).
- A storage or wire-format integrity field, or matching a device or protocol
  that mandates a specific CRC: =crc= with the matching preset (or a custom
  =crc_spec=). For a general 32-bit check, =crc32= (zlib/PNG) or =crc32c=
  (hardware-accelerated on many CPUs).

Confidence: every CRC preset is pinned to its reveng catalogue check value for
the string "123456789" and differentially validated against an independent
bit-serial CRC; the sum family is pinned to the published Adler-32 and
Fletcher-16/32/64 vectors and checked against a naive per-unit reference past
its block boundary and on partial final units.
